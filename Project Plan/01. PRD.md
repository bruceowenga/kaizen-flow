## Executive Summary

TaskFlow is a hybrid task management application designed for developers and knowledge workers who struggle with digital task systems but find physical notebooks effective. The product combines the immediacy and visibility of paper with the persistence and accessibility of digital tools, grounded in GTD, Gemba Kaizen, and 7 Habits principles.

**Core Hypothesis:** Digital task systems fail not because they lack features, but because they have too many. TaskFlow succeeds by enforcing constraints that mirror physical notebooks while adding minimal digital advantages.

---

## Problem Statement

### User Pain Points

1. **Existing digital tools are abandoned within weeks** due to complexity, notification fatigue, and organizational overhead
2. **Physical notebooks work but are fragile** - easy to lose the habit, can't sync across devices, no reminders
3. **Mental task management fails** under cognitive load, leading to dropped commitments and stress
4. **Context switching is expensive** - current tools require multiple clicks to see what's next

### Success Metrics

- **Primary:** 80%+ users still actively using after 30 days (vs. <20% for traditional tools)
- **Secondary:** Average time from task capture to completion < 48 hours
- **Tertiary:** Weekly review completion rate > 70%

## Target User

### Primary Persona: "Bruce" - The Self-Directed Developer

**Demographics:**
- Software engineers, DevOps practitioners, technical founders
- 25-40 years old
- Manages multiple contexts (coding, infrastructure, business tasks)
- Works independently or on small teams (3-10 people)

**Behavioral Characteristics:**
- Has tried 3+ task management apps and abandoned all
- Values simplicity over features
- Comfortable with CLI/keyboard shortcuts
- Runs homelab or personal infrastructure
- Prefers local-first, privacy-respecting tools

**Jobs to Be Done:**
1. Capture tasks the moment they occur without breaking flow
2. Always know "what should I do next?" without thinking
3. Prevent tasks from being forgotten without constant vigilance
4. Maintain system usage without willpower or discipline

## Product Principles

### 1. Friction Asymmetry
- **Zero friction to capture** (< 3 seconds from thought to recorded)
- **Intentional friction to defer** (must explicitly decide, can't just ignore)
- **Automatic friction to abandon** (system forces weekly reconciliation)

### 2. Radical Visibility
- **Next action always visible** without opening app
- **No hidden menus or views** - everything is one level deep
- **Spatial consistency** - things stay in the same place

### 3. Enforced Simplicity
- **One "now" task** - artificial scarcity prevents overwhelm
- **No projects or hierarchies** - flat list only
- **Three states maximum** - Now, Later, Waiting

### 4. Built-In Forgiveness
- **Expects abandonment** and makes return easy
- **No streaks or guilt** - focuses on outcomes, not consistency
- **Archive, never delete** - can always review what happened

### 5. Local-First, Cloud-Optional
- **Works offline always** - no network dependency
- **Sync via user infrastructure** - Git, not proprietary servers
- **User owns data** - SQLite files, plain text export

### F3: Weekly Review Enforcement

**Description:** Mandatory weekly task reconciliation to prevent system rot

**Trigger Logic:**
```python
if today == "Sunday" and time >= "20:00":
    if tasks_in_inbox > 0 or last_review > 7_days:
        trigger_review_flow()
        
if tasks_added_since_review > 50:
    trigger_review_flow()  # Emergency review
```

**Review Flow:**

**Step 1: Welcome Screen**
```
This week you captured 23 tasks.
You completed 15. Nice work.

8 tasks need decisions.
This will take about 5 minutes.

[Start Review] [Remind me in 1 hour]
```

**Step 2: Rapid Triage**
- Show one task at a time (full screen)
- Swipe/Arrow key interactions:
  - **Right** → Do Now (moves to NOW section)
  - **Up** → Schedule (calendar picker)
  - **Down** → Someday/Maybe (archive)
  - **Left** → Delete (with confirmation)
- Progress bar: "3 of 8 remaining"

**Step 3: Completion Summary**
```
✅ Review complete!

Actions:
- 3 tasks → NOW queue
- 2 tasks → Scheduled  
- 2 tasks → Someday/Maybe
- 1 task → Deleted

Next review: Sunday, Feb 12 at 8 PM

[Done] [Review Someday/Maybe]
```

**Review Lockout:**
- If review skipped 2 weeks consecutively:
  - Can still capture tasks
  - Cannot mark tasks complete
  - Dashboard shows: "⚠️ Review overdue - limited functionality"

**Acceptance Criteria:**
- [ ] Review flow completable in < 7 minutes for 50 tasks
- [ ] Can reschedule review time in settings
- [ ] Notifications 1 hour before, at scheduled time, and 1 hour after
- [ ] Keyboard shortcuts for all triage actions
- [ ] Progress saved - can exit and resume review

### F4: One-Now-Task Rule

**Description:** Enforced constraint limiting "NOW" section to single task

**Behavior:**
- Only one task can have "NOW" status at a time
- Attempting to mark second task as NOW:
  ```
  You're already working on:
  "Fix 3CX user adoption issues"
  
  Move this to NOW instead?
  
  [Yes - switch tasks]  [No - keep current]
  ```
- Current NOW task automatically moves to NEXT when replaced
- Timer starts automatically when task moves to NOW
- Desktop notification: "NOW task: [task name]"

**Exceptions:**
- Can force-override with `Shift+Click` (logs override for analytics)
- Review mode temporarily disables rule

**Rationale:**
Creates artificial scarcity similar to single notebook page visibility, preventing task-switching without conscious decision.

**Acceptance Criteria:**
- [ ] Rule enforced across all platforms (desktop/mobile)
- [ ] Clear visual indication of current NOW task
- [ ] Override requires explicit confirmation
- [ ] Time tracking persists across app restarts

### F5: Local-First Data Architecture

**Description:** SQLite-based storage with Git synchronization

**Database Schema:**

```sql
-- tasks table
CREATE TABLE tasks (
    id TEXT PRIMARY KEY,  -- UUID
    title TEXT NOT NULL,
    status TEXT CHECK(status IN ('now', 'next', 'waiting', 'scheduled', 'someday', 'done')),
    created_at INTEGER NOT NULL,  -- Unix timestamp
    updated_at INTEGER NOT NULL,
    completed_at INTEGER,
    scheduled_for INTEGER,  -- Unix timestamp
    context TEXT,  -- 'work', 'personal', 'infrastructure'
    notes TEXT,
    parent_id TEXT,  -- For subtasks (future)
    
    -- Time tracking
    time_spent_seconds INTEGER DEFAULT 0,
    started_at INTEGER,
    
    -- Metadata
    source TEXT,  -- 'desktop_hotkey', 'mobile_widget', 'email', 'api'
    tags TEXT,  -- JSON array
    
    FOREIGN KEY(parent_id) REFERENCES tasks(id)
);

-- reviews table
CREATE TABLE reviews (
    id TEXT PRIMARY KEY,
    started_at INTEGER NOT NULL,
    completed_at INTEGER,
    tasks_reviewed INTEGER,
    tasks_completed INTEGER,
    tasks_deferred INTEGER,
    tasks_deleted INTEGER,
    duration_seconds INTEGER
);

-- sync_log table (for conflict resolution)
CREATE TABLE sync_log (
    id TEXT PRIMARY KEY,
    timestamp INTEGER NOT NULL,
    operation TEXT,  -- 'push', 'pull', 'conflict'
    details TEXT  -- JSON
);

-- settings table
CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT
);
```

**Sync Strategy:**

**Option 1: Git-Based (MVP)**
```bash
# Every 5 minutes or on manual trigger
cd ~/.taskflow/
git add taskflow.db
git commit -m "Sync: $(date)"
git pull --rebase
git push

# Conflict resolution: last-write-wins with backup
```

**Option 2: Custom Sync (Future)**
- CRDTs for conflict-free replication
- WebSocket connection to homelab server
- Real-time sync when online

**Data Location:**
- **Linux:** `~/.local/share/taskflow/`
- **macOS:** `~/Library/Application Support/TaskFlow/`
- **Windows:** `%APPDATA%/TaskFlow/`
- **iOS:** App container (iCloud sync optional)
- **Android:** `/data/data/com.taskflow/`

**Export Formats:**
- JSON (full fidelity)
- Markdown (human-readable)
- CSV (for analysis)
- Plain text (simple backup)

**Acceptance Criteria:**
- [ ] Database operations < 50ms for typical queries
- [ ] Sync handles offline → online transitions gracefully
- [ ] Conflicts detected and user-resolvable
- [ ] Full export available from UI
- [ ] Database auto-vacuums weekly

### F6: Context Detection (Automatic)

**Description:** Intelligently filter tasks based on current context

**Detection Methods:**

**Time-Based:**
```python
if 9 <= hour <= 17 and weekday:
    context = "work"
elif hour >= 22 or hour <= 6:
    context = "personal_low_energy"
else:
    context = "personal"
```

**Location-Based (Optional, Privacy-Respecting):**
```python
if vpn_connected and vpn_name == "TSAVO":
    context = "work"
elif wifi_ssid == "HomeSSID":
    context = "personal"
```

**Activity-Based:**
```python
if active_app in ["VSCode", "Terminal", "Docker Desktop"]:
    context = "development"
elif active_app in ["Outlook", "Teams", "Zoom"]:
    context = "communication"
```

**Manual Override:**
- Dropdown in dashboard: "Work | Personal | Infrastructure"
- Hotkey: `Cmd+Shift+1/2/3` for context switching
- Shows all tasks by default, highlights relevant ones

**Context-Specific Filtering:**
- NEXT section shows only context-relevant tasks
- NOW task always visible regardless of context
- Badge shows hidden task count: "(+12 other contexts)"

**Acceptance Criteria:**
- [ ] Context detection accuracy > 85%
- [ ] Manual override persists until next detection
- [ ] Works offline (no location API dependencies)
- [ ] Clear visual indication of active context

### F8: MCP Server Integration

**Description:** Model Context Protocol server for Claude Code integration

**API Endpoints:**

```python
# claude_code_mcp_server.py

@mcp.tool()
async def task_add(title: str, context: str = "development") -> str:
    """Add a new task to TaskFlow"""
    task_id = await db.create_task(title, context=context)
    return f"Task created: {task_id}"

@mcp.tool()
async def task_complete(task_id: str) -> str:
    """Mark a task as complete"""
    await db.complete_task(task_id)
    return f"Task {task_id} completed"

@mcp.tool()
async def task_list(context: str = None, status: str = "next") -> list:
    """List tasks, optionally filtered by context"""
    tasks = await db.get_tasks(context=context, status=status)
    return [{"id": t.id, "title": t.title} for t in tasks]

@mcp.tool()
async def task_current() -> dict:
    """Get the current NOW task"""
    task = await db.get_now_task()
    return {"id": task.id, "title": task.title, "time_spent": task.time_spent}
```

**Claude Code Usage:**
```bash
# In Claude Code chat
User: "Add a task to investigate the Ghost blog SSL issue"
Claude: [Calls task_add("Investigate Ghost blog SSL issue", "infrastructure")]

User: "What am I supposed to be working on?"
Claude: [Calls task_current()]
        "You're currently working on: Fix 3CX user adoption issues (2h 15m)"

User: "Mark that done and show me what's next"
Claude: [Calls task_complete(), then task_list(status="next")]
```

**Homelab Integration:**
```python
# monitoring_alert.py
async def on_alert(service: str, severity: str):
    if severity == "critical":
        await taskflow.task_add(
            f"URGENT: {service} is down",
            context="infrastructure"
        )
```

### F9: Collaboration Lite

**Description:** Minimal task sharing for small teams

**Constraints:**
- No real-time collaboration
- No comments or @mentions
- No task dependencies
- Maximum 10 shared lists per user

**Features:**

**Shared Lists:**
```
TSAVO - Infrastructure Tasks (shared with 3 people)
  → Upgrade Docker on Odin
  → Fix Heimdall SSL cert
  [Assigned to: Bruce]
  
  → Set up monitoring for MediaCMS
  [Assigned to: John]
```

**Assignment Flow:**
1. Create shared list: "TSAVO - Infrastructure"
2. Invite members via email (generates unique link)
3. Assign task → creates entry in their inbox
4. They accept → moves to their NEXT queue
5. They complete → updates in shared list

**Permissions:**
- **Owner:** Can add/remove members, delete list
- **Member:** Can add tasks, assign to others, complete own tasks

**Sync:**
- Shared lists sync via central server (not Git)
- Local tasks remain local-first
- Clear visual distinction between personal and shared
